Project Highlights

Custom Solution to Distribute Records Instead of Omni-Channel (Soon to be on AppExchange)

Requirements:

  • Departments are in need to prioritize certain case records based on defined criteria.
  • Users should have a component to see which records are assigned to them with fields and their values, similar to Omni-Channel component.
  • Users should be able to view a record which should be directed from the component.
  • The component should paginate assigned records in 5.
  • The criteria and their priority should be modifiable by managers.
  • Record assignments should be stoppable by managers.
  • Standard Queues and Public Groups should be used to decide to whom records will be distributed to.

Solution:

  • Create 2 custom objects (Parent) Round Robin Rule, (Child) Round Robin Query Sort respectively.
  • Have a checkbox for managers to stop users to own a record on the parent object.
  • Have a text field to specify queue developer name for managers to decide which users should be able to own records on the parent object.
  • Have a text field to place a query that returns certain amount of records with specified criteria on the child object.
  • Have a number field to specify the prioritization for the managers on the child object.
  • Create a LWC component and place it in the utility bar in the app.
  • Place a "Own a Record" button on LWC. When a user clicks the button, query all child records in an ascending order based on priority and query records.
  • Assign the first record to user.
  • I can't explain how the rendering happens because it's my intellectual property.

Challenges:

  • Queues have more than 90 users and it's likely that they might click to the button and the record is assigned to both users within miliseconds one after the other. Record locking was needed to prevent it.
  • Since the first record is locked for the whole transaction, users might have gotten the "record is locked error" or "record is owned by someone else, try again" that was thrown. randomization was introduced meaning that if the rule returned 20 records in total based on priority, one of the first 5 records was randomly picked.

Close Case API and Refactoring

Context:

  • Users close different types of cases either successfully or unsuccessfuly using a screen flow.
  • They need to choose one mandatory and two optional unsuccessful reasons, and a comment.
  • The screen flow, closes the case and based on the type, updates different records.

Requirements:

  • An external system needs to be able to close a certain type of case unsuccessfully which is a process that's already exists and is triggered via screen flow.
  • The endpoint and screen flow should follow the same logic.
  • When a case is closed through endpoint, the owner should be changed to user in salesforce with the same azure id that's provided in the request.

Solution:

  • Create a class utilizing factory pattern so if the same or other external systems require to close cases successfully or unsuccessfully, we don't create a new class each time.
  • Create an invocable method and replace the logic in screen flow with it so the record can be locked to prevent overwriting.

Challenges:

  • Owner of the case should only be changed when it's closed through endpoint and not invocable method, created a wrapper to specify whether owner change is needed or not along with other values.
  • External system might send rejection reasons between 1 or 3, dynamic apex is used respected fields are assigned to provided values instead of bunch of repetitive if and else blocks.
  • Other related records were updated on the case, there were lots of possibilities for things to go south. Custom exception classes are created to throw exceptions between classes to warn other systems and users who are using the screen flow.

Reassignment of Sales Calls via Rest API

Context:

  • Users have two types of sales calls as records in event object, reserved and available slot.
  • Booking agents and an external system are responsible for booking available slots and turn them into reserved ones.
  • External system provides event id that needs to be reassigned and azure id of the targeted assignee.
  • Reserved events have the following naming convention for subject: customer name - user name.

Requirements:

  • Targeted user has an available slot at the same start date/time needs to be checked.
  • The event id that is sent and the available slot of targeted assignee should be locked so other systems or users shouldn't overwrite the current transaction.
  • Records that are related to what polymorphic relationship should be updated such as what id's (case) parent case, opportunity and other child cases.
  • The naming convention of the reassigned event's subject needs to be modified; targeted assignee's name should be replaced with the previous one, and "(Reasigned)" needs to be added at the end of it.
  • If all criteria met, the available slot of the target assignee should be deleted.

Solution:

  • Created an endpoint for the external system to make a call out with two parameters: event id and azure id of the targeted user.

Challenges:

  • TYPEOF is used to query case in what id polymorphic lookup along with its parent case and opportunity to save two SOQL statements.
  • All records to be updated are collect into SObject list to make transaction control easy when an exception is caught.

Global Sales

Context:

  • There are a few sales channels where leads are received.
  • They're all following the same customer journey with small changes: owners are not the same for leads, parent case, opportunity, account and child cases. Some child cases needs to be auto-closed (inserted as Closed - Successful). Leads need to be auto-converted excluding a criteria for some channels and for some this criteria should be there.
  • These rules were in different flows and when a new sales channel is introduced, we needed to change all of the flows, resulting in 5-7 user stories.

Requirements:

  • Map all sales channels as key to its corresponding lead, account, opportunity, parent case owners, auto-conversion criteria exclusion option in custom metadata.
  • Apply this logic to already existing customer journey logic which is managed in case triggers that is responsible for creating new cases.

Solution:

  • Reference the custom metadata in all flows and triggers so when a new sales channel comes, all it's needed is to add a new entry to custom metadata.

Challenges:

  • There is already an existing logic which is in case trigger to make sure customer journey proceeds. Global sales needed to be applied to it instead of extending the logic and making it more complicated.

Anonymize Leads

Context:

  • Companies in Europe needs to comply with GDPR as a result lead anonymization project was initiated.
  • We might have multiple leads for the same customer or a customer might already have an account.
  • External id is used to identify a customer

Requirements:

  • Personal identification fields (such as First Name, Last Name, Email etc.) of Lead records that are lost or not updated for certain time needs to be anonymized, a default value needs to be placed.
  • Default values for each personal identification field needs to be mapped in a custom metadata.

Solution:

  • Create a batch and schedule apex to query leads and anonymize them.

Challenges:

  • Other systems subscribe to changes of personal identification fields, we need to be absolutely sure to anonymize a lead or not.
  • If there is any other newer lead that a customer has or if a customer has already have an account, anonymization for that customer shouldn't be executed.
  • If there is no newer leads and no accounts for the customer, all leads that belong to the customer needs to be anonymized.
  • All execution a customer from anonymization and anonymization processes need to be done in a bulkified way.
